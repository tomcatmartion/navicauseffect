# 性格分析模块最终实现指南（融合 Gemini 方案）

Gemini 提出的“4个后台函数 + JSON 组装 + 大模型翻译”方案，与我们之前确定的“确定性计算 + 大模型表达”架构完全一致，且给出了非常具体的落地细节。以下是经过评审和补充后的**最终实现方案**，可直接交给 Claude Code 编码。

---

## 一、整体流程

```
FinalIR（含评分、四维标签、格局）
       ↓
[1] 空间定位函数 → 获取命宫、身宫、太岁宫的本对合临四维星曜
       ↓
[2] 能量加权函数 → 决定各宫位的主导权重，生成标签（如“外强中干”“羊陀夹忌”）
       ↓
[3] 知识提取函数 → 根据星曜+旺弱从知识字典提取关键词
       ↓
[4] JSON 组装器 → 构建结构化性格 IR（心理学中间态）
       ↓
大模型（唯一调用）→ 将 IR 翻译为兄长风格的自然语言报告
```

所有步骤中，**只有最后一步依赖大模型**，前四步全是确定性代码。

---

## 二、四个后台函数详解

### 2.1 函数一：`getFourDimensions(palaceName, chartData)`

**职责**：给定一个宫位名称（如“命宫”），返回该宫位的**本宫、对宫、三合宫（两个）、夹宫（左右邻宫）** 的星曜数据。

**输入**：
- `palaceName`: string
- `chartData`: BaseIR（包含十二宫的完整信息）

**输出**：
```typescript
{
  ben: PalaceData,           // 本宫
  dui: PalaceData,           // 对宫
  sanHe: [PalaceData, PalaceData], // 三合宫（两个）
  lin: { left: PalaceData, right: PalaceData } // 左右夹宫
}
```

**实现要点**：
- 利用固定地支顺序：`['子','丑','寅','卯','辰','巳','午','未','申','酉','戌','亥']`
- 对宫索引 = 本宫索引 + 6（模12）
- 三合：本宫索引 + 4 和 +8（模12）
- 夹宫：本宫索引 -1 和 +1（模12）
- **注意**：夹宫即使无主星，也要返回（空宫），但后续加权时会判断是否构成“夹”。

**补充规则**：对于身宫、太岁宫，同样适用此函数。

---

### 2.2 函数二：`calculatePersonalityWeight(palaceData, fourDimData)`

**职责**：根据宫位的最终评分（finalScore）、亮度、四维合参标签，决定性格的主导方向和权重。

**输入**：
- `palaceData`: 本宫的评分、主星、亮度、四化等
- `fourDimData`: 该宫位的四维合参结果（来自之前的四维合参模块）

**输出**：一组结构化标签，例如：
```json
{
  "dominance": "benGong",      // 主导权：benGong / duiGong / sanHe / linGong
  "special_patterns": ["羊陀夹忌"],  // 特殊夹格
  "personality_bias": "外柔内刚",   // 综合定性
  "weight_notes": ["命宫弱，对宫强 → 容易受外界影响"]
}
```

**实现逻辑**：
- 如果本宫finalScore ≥ 6.0，则本宫主导。
- 如果本宫 < 4.0 且对宫 ≥ 6.0，则标记“外强中干”或“靠外部环境”。
- 如果夹宫中有擎羊+陀罗且本宫有化忌，强制添加“羊陀夹忌”标签。
- 如果左右邻宫分数差距 > 2，标记“夹宫失衡”。
- 以上规则可以直接写在代码中（if-else），因为数量有限。

---

### 2.3 函数三：`fetchPersonalityTraits(starName, brightness, palaceRole)`

**职责**：从知识字典中提取该星曜在指定亮度下的性格关键词。

**输入**：
- `starName`: string
- `brightness`: "旺"|"平"|"陷"|"实旺"|"磨炼"|"虚浮"|"凶危"
- `palaceRole`: "命宫"|"身宫"|"太岁宫"|"对宫"|"夹宫"

**输出**：字符串数组，如 `["热情", "外向", "固执"]`

**实现**：
- 加载 `star_palace_dict.json`，key 格式为 `{星曜}_{亮度}`（如 `天同_旺`），value 为关键词数组。
- 如果亮度是“实旺”或“磨炼”，统一映射到“旺”；“虚浮”“凶危”映射到“陷”。
- 对于特殊组合（如天机+天梁同宫），可以在 JSON 中增加组合键 `天机天梁_旺`，优先匹配。

---

### 2.4 函数四：`buildPersonalityIR(finalIR, fourDimResults, traits, weights)`

**职责**：将以上所有结果组装成一个**心理学中间态 JSON**，供大模型使用。

**输出结构**（参考 Gemini 示例，稍作扩展）：

```json
{

  "layers": {
    "ming_palace": {
      "role": "表层_外在表现",
      "dominant_stars": ["天同(旺)", "天梁(陷)"],
      "four_dimension": {

        "san_he_support": "强力",
        "lin_gong_status": "左强右弱",

      },
      "keywords": ["温和", "随和", "知足", "有担当", "略固执"],
```

## 三、大模型 Prompt 设计

### System Prompt（固定）

```
你是紫微斗数性格分析师，语气温暖如兄长。不要直接罗列星曜名称，而是将性格特质转化为现代心理学的语言（如“防御机制”“高敏感”“控制欲”等）。严格基于提供的 JSON 数据，不要自行添加外部知识。
```

### User Prompt 模板

```
【性格诊断书】
{personalityIR}

【任务】
请以「你」称呼用户，写一段 250-350 字的性格分析。结构要求：
1. 先说表层性格（命宫），让用户觉得“说得真准”。
2. 再转折，点出他中层（身宫）和内核（太岁宫）的矛盾或不易，给与理解与抚慰。
3. 最后提一下整体底色（如化忌、格局），给一个正向的鼓励或建议。
语言要像朋友聊天，不要像算命先生。
```

## 四、与现有架构的整合点

| Gemini 函数 | 对应我们已有模块 | 整合方式 |
|-------------|-----------------|----------|
| 空间定位 | 新增 `lib/fourDimension.ts` | 独立函数，供性格分析调用 |
| 能量加权 | 复用四维合参标签生成器 | 直接使用 FinalIR 中的 `fourDimensionTags`，无需重写 |
| 知识提取 | `KnowledgeDict.getStarAttr()` | 已有，直接调用 |
| JSON 组装 | `buildPersonalityIR()` | 新增函数，放在 `lib/personalityBuilder.ts` |
| 大模型调用 | `aiClient.chat()` | 已有，复用 |

**新增文件**：
- `lib/fourDimension.ts` — 实现 `getFourDimensions` 函数
- `lib/personalityBuilder.ts` — 实现 `buildPersonalityIR` 函数，内部调用知识提取和四维数据

**修改文件**：
- `orchestrator.ts` — 在 `FINAL_IR_READY` 状态后，自动调用性格分析流程：
  ```typescript
  const personalityIR = buildPersonalityIR(finalIR);
  const prompt = buildPersonalityPrompt(personalityIR);
  const report = await aiClient.chat(prompt);
  session.personalityReport = { ir: personalityIR, text: report };
  ```

## 五、测试用例

准备 3 个真实命盘（不同性格类型），手工计算性格关键词和 IR，然后对比大模型输出是否一致。

**验收标准**：
- 大模型不会说出 JSON 中不存在的内容。
- 不会出现“根据星曜，您…”这种机械句式。
- 语气符合兄长风格，有共情。

## 六、给 AI开发工具的 的开发指令

> 
>
> 请按照以下要求实现性格分析模块：
>
> 1. 实现 `getFourDimensions(palaceName, baseIR)` 函数，返回本对合临四宫的星曜数据（使用地支固定偏移）。
> 2. 实现 `buildPersonalityIR(finalIR)` 函数，输出如上所述的 JSON 结构。其中：
>    - 命宫、身宫、太岁宫各自的 `keywords` 通过 `KnowledgeDict.getStarAttr` 获取。
>    - 四维合参标签直接从 finalIR 中的 `fourDimensionTags` 读取。
>    - 特殊标签（如“外柔内刚”）根据命宫与身宫分数对比生成。
> 3. 实现 `buildPersonalityPrompt(personalityIR)` 函数，组装大模型 Prompt。
> 4. 在编排层中，当 finalIR 生成后，自动调用上述流程生成性格报告并存入 session。
> 5. 编写单元测试，验证 `getFourDimensions` 在 12 个不同命宫位置下返回正确的四宫。

建议：
 给开发落地的两点“防坑”微调建议
方案的骨架已经极度完美，但在让AI工具动手写代码之前，可以再补充两个命理学特有的业务约束，防止代码在极端盘面下报错：

1，“空宫借星”的硬逻辑拦截： 在编写第一步的 generatePersonalityTags 时，必须让代码优先判断命宫或身宫是否为“空宫（无主星）”。如果是空宫，代码必须强制去抓取对宫（迁移宫等）的主星，并自动降低一个亮度等级存入 JSON，同时自动追加一个 ["容易受外界环境影响", "缺乏核心主见"] 的底层标签。绝不能把空数据直接抛给大模型。

2，负面标签的“心理学强制包装”： 虽然 JSON 里会直白地输出风险点（如 ["想做成事但总遇到阻碍"]），但在构建发给大模型的 Prompt 时，建议强行加入一条指令：“当遇到风险提示时，必须采用心理疏导的视角，将其解释为一种『性格防御机制』或『可以转化为优势的特质』，严禁使用铁口直断的负面判词。”

3，“六吉六煞”不能单看，必须【组合取象】
文档风险点：大模型可能会简单地翻译“紫微=领导，左辅=助力”，但紫微斗数讲究“君臣庆会”或“孤君在野”。

开发指令：在后台提取标签时，加入一个主辅星化学反应检测（Combination Check）。如果命宫主星是紫微，代码必须扫描三方四正有没有左辅/右弼。如果有，输出标签 ["有领导力且得人心", "君臣庆会"]；如果没有，输出标签 ["主观极强", "高处不胜寒", "孤君"]。这种组合判断必须写在代码的 JSON 逻辑里，不能指望大模型自己去“顿悟”。


请确保所有函数都有明确的 TypeScript 类型定义，并添加必要的注释。完成后，集成到主系统中。
