## 需要修正的问题

### 问题一：M2 评分引擎的复杂度仍被低估

文档说 `scoring.ts` 约 150 行，但六步评分法里有几个隐藏复杂度：

**衰减系数不是固定的**，`attenuation.json` 需要存储的是一个二维矩阵（本宫旺弱等级 × 目标宫旺弱等级的组合），不是几个简单数字。这个 JSON 的设计本身就需要仔细规划。

**建议**：Sprint 2 开始前，先把 `attenuation.json` 的数据结构设计出来给 CC 确认，避免中途返工。

---

### 问题三：PromptBuilder 的"命盘速览"需要明确格式

文档给的示例很好，但需要规定哪些宫位必须出现在速览里，哪些按需出现。否则 CC 写出来的速览可能每次不一致。

**建议**：固定速览格式如下：

```
始终出现（每轮都在）：
  命宫、大限命宫、流年命宫 + 对应四化
  用户当前关注的主看宫位 + 兼看宫位
  原局特殊叠加（双忌/权忌交冲等）

按需出现：
  其他宫位（只有分析涉及时才加入）
```

---

### 问题四：patterns.json 的 DSL 设计是最大的开发风险

文档里这个部分写得最轻描淡写，但实际上是整个项目最容易卡住的地方。

**原因**：84 条格局，每条条件组合各不相同，`evaluateCondition` 函数需要支持的原子条件类型可能有 10-15 种，这个"规则引擎解释器"的工作量不亚于重新写一个小型语言运行时。

**我的建议**：Sprint 2 先只实现 **10 个最重要的格局**（比如府相朝垣、双禄夹命、紫府同宫等），跑通流程后再逐步添加。不要一开始就试图把 84 条全部实现，那会导致 Sprint 2 无限延期。

## 一个缺失的模块

文档里没有提到的是**对 iztro JSON 字段的防御性处理**。

iztro 是开源项目，字段命名和数据结构版本之间可能有差异。建议在 ChartBridge 里加一个 **iztroAdapter** 层：

```typescript
// lib/iztroAdapter.ts
// 职责：标准化 iztro 的输出，屏蔽版本差异
// 如果 iztro 升级了字段名，只需要改这一个文件

function adaptPalaces(raw: any): NormalizedPalace[] { ... }
function adaptSihua(raw: any): NormalizedSihua { ... }
```

这 30 行代码能避免未来 iztro 升级时全局改动。
