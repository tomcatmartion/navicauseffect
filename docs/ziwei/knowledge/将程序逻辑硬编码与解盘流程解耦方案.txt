### 3.4 评分流程伪代码（`scoring-flow.ts`）

```typescript
export function scorePalace(palace: PalaceData, params: ScoringParams, chart: Chart): ScoredPalace {
  let score = getSkeletonScore(chart.skeletonId, palace.dizhi);
  const maxScore = params.initial_ceiling[palace.initialBrightness];

  // 加分（吉星 + 化禄）
  score += sumStars(palace, params.star_scores, '本宫') * 1.0;
  score += sumStars(palace.opposite, params.star_scores, '对宫') * params.decays.opposite;
  // ... 三合、夹宫类似

  // 重定亮度
  const brightness = determineBrightness(score, params.thresholds);

  // 减分（煞星）—— 根据新亮度查衰减系数
  const badDecay = getBadStarDecay(brightness, params);
  score += sumBadStars(palace, params.star_scores, '本宫') * badDecay;
  // ... 对宫、三合、夹宫类似

  // 禄存处理
  if (palace.hasLuCun) score += params.lu_cun_delta[brightness];

  return { ...palace, finalScore: Math.min(score, maxScore), brightness };
}
```

### 3.5 格局匹配器（`pattern-matcher.ts`）

实现递归条件求值函数 `evaluateCondition(cond, chart)`，支持 `and/or/not` 以及原子条件类型：
- `star_in_palace`
- `star_in_triples`
- `any_star_in_triples`
- `all_stars_in_triples`
- `palace_dizhi`
- `palace_brightness`
- `special_structure`（如“羊陀夹忌”需额外实现检测函数）

匹配结果写入 `FinalIR.patterns`。

四、程序如何读取并执行
启动时：config-watcher 加载所有 data/ 下的 JSON 文件到内存。

运行时：scoring-flow.ts 从内存中的 scoringParams 对象读取系数；pattern-matcher.ts 从 patterns 数组读取格局条件并求值；router 从 routerTree 对象读取决策树；等等。

热加载：您修改任意 JSON 文件后，fs.watch 触发重载，下次调用相应模块时自动使用新值，无需重启 Node.js 进程。

示例：您想调整“旺”宫的亮度阈值从 7.5 改为 7.0。编辑 scoring-params.json 中的 "旺": 7.0，保存。下一次用户请求时，评分函数会读到新阈值，评分结果立即变化。不需要重新编译或部署代码。

最终答案：是的，所有 SKILL_ 文档中可变动的规则、参数、取象文本、问诊树都已迁移到 data/ 目录的 JSON 文件中。以后您的解盘规则、流程、技法发生变化时，只要变化的内容属于上述“可配置”范畴，就只需修改 JSON 文件，程序会动态读取并生效。只有极少数底层算法（如四维合参的计算公式）仍保留在代码中，但那些通常是百年不变的命理基础，不需要您频繁调整。
